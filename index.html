ARM 汇编常用寻址方式

发表于 2016-10-02   |   分类于 学习 ， 嵌入式   |  
  1. 寄存器直接寻址
    操作数的值直接放在寄存器中，指令中的地址码字段指出的是寄存器编号，指令执行时直接取出寄存器值来操作。
    例如，MOV R1,R2
  2. 寄存器移位寻址
    例如，操作数 2 先进行位移操作，然后再与第一个操作数进行操作。
    MOV R0,R1,LSL #3，这条指令将 R1 的值左移三位，然后将结果放在 R0 中。
  3. 寄存器间接寻址
    指令中的地址码给出的是一个通用寄存器的编号，指令所需的操作数保存在寄存器指定地址的存储单元中，即寄存器为操作数的地址指针。
    例如，LDR R1,[R2]，这条指令将 R2 指向的存储单元中的数据读出保存在 R1 中。
  4. 基址寻址
    将基址寄存器的内容与指令中给出的偏移量相加，形成操作数的有效地址，常用于查表和数组操作。
    例如，LDR R2,[R3,#0X0C]，此语句将 R3+0X0C 处的数据存放到 R2 寄存器中。
  5. 立即寻址
    立即寻址中，操作码字段后面的部分不是操作数地址而是操作数本身。
    例如，MOV R1,#0XAA，此语句将立即数 0XAA 存放到寄存器 R0 中。

补充：叹号的用法

  1. pre-indexed addressing，先进行方括号中的操作并写回方括号中的第一个操作数

  2. post-indexed addressing，先赋值，再操作

U-Boot 相关前置知识

发表于 2016-09-30   |   分类于 学习 ， 嵌入式   |  

Bootloader

能将操作系统内核复制到内存中运行，这段程序为 Bootloader。
Bootloader 是一段小程序，在系统上电时开始执行，初始化硬件设备，准备好软件环境，最后调用操作系统内核。

Bootloader 的结构和启动过程

概述

嵌入式 Linux 系统从软件的角度通常可以分为以下4个层次。

  1. 引导加载程序，包括固件中的 boot 代码和 Bootloader。 x86 会先运行 BIOS 中的固件，然后才运行硬盘上的第一个分区（MBR）中的 Bootloader。大多数嵌入式系统中没有固件，Bootloader 是上电后执行的第一个程序；
  2. Linux内核。 特定于嵌入式板子的定制内核以及内核的启动参数。此参数可能是内核默认的，也可能是 Bootloader 传递给它的；
  3. 文件系统。 包括根文件系统和建立于 Flash 内存设备上的文件系统；
  4. 用户应用程序。

阶段

  1. 第一阶段

    • 对硬件初始化*；
    • 准备加载 Bootloader 的第二阶段代码准备 RAM 空间；
    • 复制 Bootloader 的第二阶段代码到 RAM 空间中**；
    • 设置好栈；
    • 跳转到第二阶段代码的 C 入口点。
      * 包括关闭 WATCHDOG、关中断、设置 CPU 的速度和时钟频率、RAM 初始化等。这些都不是必须的，比如 S3C2440 的开发板所使用的 U-Boot 中，就将 CPU 的速度和时钟频率的设置放在第二阶段。
      ** 不是必须的，对于 NOR Flash 等存储设备，可以在上面直接执行代码，但是效率比在 RAM 中执行大为降低。

  2. 第二阶段

    • 初始化本阶段要使用的硬件；
      至少要初始化一个串口以便程序员与 Bootloader 进行交互。
    • 检测系统内存映射（Memory Map）；
      确定板上使用了多少内存、它们的地址空间是什么。根据板子直接设置，而不用写适配各种情况的复杂算法。
    • 将内核映像和根文件系统映像从 Flash 上读到 RAM 空间中；
      将根文件系统映像复制到 RAM 中，这不是必须的。
    • 为内核设置启动参数；
    • 调用内核。

  3. 后续工作
    将内核存放在适当的位置后，直接跳到它的入口点即可调用内核。调用内核之前，下列的条件要满足。

    1. CPU 寄存器的设置
      • R0 = 0；
      • R1 = 机器类型 ID；参见linux//arch/arm/tools/mach-types。
      • R2 = 启动参数标记列表在 RAM 中起始基地址。
    2. CPU 工作模式
      • 必须禁止中断（IRQs 和 FIQs）；
      • CPU 必须为 SVC 模式。

        管理模式(SVC)，操作系统保护代码，系统复位和软件中断响应时进入此模式。

    3. Cache 和 MMU 设置
      • MMU 必须关闭；
      • 指令 Cache 可以打开也可以关闭；
      • 数据 Cache 必须关闭。

Bootloader 与内核的交互

Bootloader 与内核的交互是单向的，由 Bootloader 将参数传递给内核。由于它们不能同时运行，所以 Bootloader 将参数放在某个约定的地方之后，再启动内核，内核启动后从这个地方获取参数。
　　
除了约定地址外，还要规定参数的结构。Linux 2.4.x 之后的内核都期望以标记列表（Tagged List）的形式来传递启动参数。标记，是一种数据结；标记列表就是挨着存放的多个标记。标记列表以标记 ATAG_CORE 开始，以标记 ATAG_NONE 结束。

标记的数据结构为 tag，它由一个 tang_header 结构和一个联合(union)组成。tag_header 结构表示标记的类型和长度，比如类型就包括内存还是命令行参数。对于bu不同的联合（union），比如表示内存时使用 tag_mem32，表示命令行时使用 tag_cmdline。数据结构 tag 和 tag_header 定义在 Linux 内核源码的 include/asm/setup.h （对于 linux-4.7.5 版本， 此文件在 \arch\arm\include\uapi\asm\setup.h）头文件中。

union 维护足够的空间来置放多个数据成员中的“一种”，而不是为每一个数据成员配置空间，在union 中所有的数据成员共用一个空间，同一时间只能储存其中一个数据成员，所有的数据成员具有相同的起始地址。关键点：共享一部分内存。

U-Boot 分析与移植

U-Boot 源码结构

U-Boot 源码根目录可以分为4类。

  1. 平台相关的或开发板相关的；
  2. 通用的函数；
  3. 通用的设备驱动程序；
  4. U-Boot 工具、示例程序、文档。
目录 特性 解释说明
board 开发板相关 对应不同配置的电路板（即使 CPU 相同）
cpu 平台相关 对应不同 CPU，在它们的子目录下仍可以进一步细分
lib_i386类似 同上 某一架构下通用的文件
include 通用的函数 头文件和开发板配置文件，开发板的配置文件都在 include/config 目录下
lib_generic 同上 通用的函数库，如 printf 等
common 同上 通用的函数，多是对下一层驱动程序进行进一步封装
disk 通用的设备驱动程序 硬盘接口程序
drivers 同上 各类具体设备的驱动程序，基本上可以通用，他们通过宏从外面引入平台/开发板相关的函数
dtt 同上 数字温度测量器或者传感器的驱动
fs 同上 文件系统
nand_spl 同上 NAND 启动支持
net 同上 各种网络协议
post 同上 上电自检程序
rtc 同上 实时时钟的驱动

U-Boot 的配置、编译、连接过程

配置

针对不同的开发板，了解它使用了哪些文件、哪个文件首先执行、可执行文件占用内存的情况，最好的方法就是阅读它的 Makefile。
编译 U-Boot 主要生成生成三个文件，U-Boot.bin（直接烧入 ROM、NOR Flash的文件），U-Boot（ELF 格式的可执行文件） 和 U-Boot.srec。同时，编译成功后，还会在 tools 子目录下生成一些工具，比如 mkimage 等。将它们复制到 /usr/local/bin 目录下，以后就可以直接使用它们了，比如编辑内核时，会使用 mkimage 来生成 U-Boot 格式的内核映像文件 uImage。

U-Boot 还没有类似 Linux 一样的可视化配置界面（如 make menuconfig），要手动修改配置文件 include/config/<board_name>.h 来裁剪、设置 U-Boot。
配置文件中有以下两种宏，一类是选项（Options），前缀为 CONFIG_，用于选择 CPU、SOC、开发板类型，设置系统时钟、选择设备驱动等。另一类是参数（Setting），前缀为 CFG_，它们用于设置 malloc 缓冲池的大小、U-Boot 的提示符、U-Boot 下载文件时的默认加载地址、Flash 的起始地址等。
U-Boot 中几乎每个文件都被编译和连接，但是这些文件是否包含有效的代码，则由宏开关来设置。

编译和连接

  1. 首先编译 cpu/$(CPU)/start.S，对于不同 CPU，还可能编译 cpu/$(CPU) 下的其他文件；
  2. 之后，对于平台或开发板相关的每个目录/每个通用目录都使用它们各自的 Makefile 生成相应的库；
  3. 将 1、2 步生成的 .o、.a 文件按照 board/$(BOARDDIR)/config.mk 文件中制定的代码其实地址、board/$(BOARDDIR)/U-Boot.lds 连接脚本进行连接；
  4. 第 3 步得到的是 ELF 格式的 U-Boot，后面 Makefile 还会将它转换为二进制格式、S-Record 格式。

Back to the Netherlands

八月二十二号回的荷兰，明天就正好一个月了。回国前心心念念暑假这两个月要好好学习，毕竟毕业就要找工作，结果是暑假完完全全荒废玩掉了。现在后悔也没有用了，不过要是两个月利用起来的话，可以学习很多知识。
暑假这两个月也没去多少地方，回广西老家花了大半个月。陪爸妈去广西北海和涠洲岛玩了五天。之后也没有出去旅游，基本都是和高中同学在南京厮混。对于南京这座城市感觉有千千万万的不舍，毕竟是非常熟悉的故乡，哪里好吃哪里好玩都是了熟于心。

说说呆了一年多的荷兰吧，我的大学在一个僻静的小城市，恩斯赫德。虽说这座城市的名头是荷兰东部最大的城市，但是跟国内城市比起来只能算个村。想在回想起来一年来的点点滴滴觉得过得太快了。硕士第一年就是上课，挣满六十学分，但是也是非常忙。荷兰的大学三个月一个学期，所以一年有四个学期，每个学期平均要学三门课，这就意味着上三个月就要考一次期末考试。说实话一开始真的很担心考试能不能过，后来才发现亚洲学生真的在学习方面有天赋，完全不用担心考试不过。
荷兰这里的大学分为两类，一类是H类，重视动手实践，以就业为导向的学校；一类是U类，重视研究能力，以学术为导向。我们大学是U类大学，但是其实我个人更想去H类大学。经过一年的学习发现U类学的都是非常高层的知识，比如软件架构，建模对性能进行分析等等非常抽象的课程，感觉并不是以就业为导向，所以想要能找到工作的话，还是要靠自学。但是回想起自己大四刚毕业那时的什么都不懂，到现在能对计算机软件硬件有更加感性的认识，也可以说这些课程也没有白上吧。

Nowadays

从九月一号开始我就开始了为期三个月的实习。这个实习是包括在两年的硕士项目里的，当初也是看好这个机会所以选择了荷兰。实习公司是大学所在城市的一家研发医疗用品的科技公司。感觉公司牛掰的地方是那些大型的精密的手术用的医疗器械，然而我的实习跟这些高端的产品没什么关系，我的题目是为一个无线血压计搭建软件平台。
具体用到的技术是在嵌入式Linux下的Qt框架开发，用的比较多的就是Qt的多线程。软件的工作流程比较明晰，从三种传感器采集数据，经过算法处理后加密，通过WiFi或蓝牙传到平板电脑的客户端软件中。平板上的软件开发是由其他公司负责，我的主要任务就是对 ARM 处理器编程，使其能够完成上述功能。
项目到现在经历了三周，这三周的工作基本上就是对软件的架构进行设计。我对这块并不是很感兴趣，因为比较喜欢动手的操作。但是也感到软件架构这个东西对成品的质量起着关键性的作用，所以也想使框架更加完善、清楚。但是深感面向对象思想并不是那么容易应用到实际当中的，要想把它运用得准确同时高效，还是需要很多开发经验的。
下周要向软件组里的同事展示我的架构，希望可以蒙混过关，赶紧进入实际实现的环节。
BTW，现在睡眠真的太缺乏了，去年有时候没课都能睡到中午，现在每天就只能睡七八个小时，中午还不能睡。很多人都说我憔悴了，哎。虽然每天起床的时候都很痛苦，想到的第一件事就是今天一定要早睡，结果每天还是不能在十点半之前躺上床。一定要慢慢改过来。

Old Friends

上周末去了阿姆斯特丹玩耍，题图就是在阿姆北边的高楼上拍的，见了去年经常一起玩的同学。去年我们玩得好的那一帮同学大概有七八个，但是今年要么是因为EIT项目去了别的学校，要么就是去了外地实习，现在还在身边的就剩一个室友。回想起去年的时光，心里还是非常不舍的。时不时的深夜饮酒畅谈，时不时的火锅聚会，还有大家一起去周边国家旅游，好像生活中很大的乐趣都来自于这些活动。今年还不知道怎么样，不管怎么样，应该不会有去年那么开心了。朋友走了，要面临毕业就业的压力了。人在过得不那么开心的时候对过往会更加怀念。
不过幸好我还是个乐观的人，现在唯一的办法就是大步向前走吧，朋友们，珍重吧。

Plan

这一年应该是人生中十分关键的一年。毕业，就业，两件大事摆在面前。说实话当初选择出国读研的动机就是暂缓一下就业压力，而现在已经没有办法回避了，只能硬着头皮上了。

加油吧少年。
